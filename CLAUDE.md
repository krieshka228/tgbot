# CLAUDE.md — заметки по проекту (для ассистента)

Этот файл — справочник по проекту для быстрого восстановления контекста.
Прочитай его целиком перед тем, как лезть в код — это быстрее и надёжнее,
чем заново просматривать все файлы.

## Что это за бот

Telegram-магазин на python-telegram-bot v20+ (async) + SQLAlchemy (async,
SQLite по умолчанию, можно Postgres через asyncpg). Товары публикуются
постами в Telegram-канале, бот их парсит и складывает в БД, покупатели
просматривают каталог в личке бота, оформляют заказ, присылают чек,
админ подтверждает оплату и выгружает отчёты в Excel.

Два способа заказать товар:
1. **Через каталог в боте** (`catalog.py` → категории → подкатегории →
   карточки товаров → кнопка «Заказать»).
2. **Через комментарий в группе обсуждения канала** (`posts.py
   handle_comment`) — покупатель пишет количество в комментариях к посту,
   бот пишет ему в личку с подтверждением (`porder:confirm:...`).

Оба пути сходятся в одну и ту же корзину/заказ (`Order` + `OrderItem`).

## Карта файлов

- `bot/main.py` — точка входа, регистрация хендлеров, джоба напоминаний.
- `bot/config.py` — переменные окружения (BOT_TOKEN, ADMIN_USER_ID,
  ADMIN_CHAT_ID, CHANNEL_ID, DISCUSSION_GROUP_ID, DATABASE_URL).
- `bot/db.py` — модели (User, Product, Order, OrderItem, BotSetting,
  PendingOrder) + вся работа с БД + кэш каталога (см. ниже).
- `bot/cache.py` — простой in-memory TTL-кэш для каталога.
- `bot/keyboards.py` — все инлайн/реплай-клавиатуры.
- `bot/utils.py` — escape_markdown, парсинг постов канала, форматирование
  корзины/заказа, парсинг количества из текста.
- `bot/excel_reports.py` — выгрузка отчётов (заказы за месяц, клиенты).
- `bot/reminders.py` — напоминания об оплате неоплаченным заказам.
- `bot/handlers/start.py` — /start, главное меню, согласие на обработку данных.
- `bot/handlers/catalog.py` — просмотр каталога покупателем (категории →
  подкатегории → товары), запуск поиска (состояние выставляется здесь,
  обрабатывается в fsm_inputs.py).
- `bot/handlers/cart.py` — корзина: показ, удаление позиций, чекаут
  (списание остатков, переход в pending).
- `bot/handlers/checkout.py` — приём чека, подтверждение/отклонение оплаты
  админом, отмена заказа покупателем.
- `bot/handlers/fsm_inputs.py` — **центральный диспетчер текстовых
  сообщений** по состояниям (`context.user_data['state']`): ввод
  количества, телефона, адреса, поиск по названию/артикулу, админская
  синхронизация товара из поста, ручное изменение остатка, массовое
  удаление по артикулам, обращение к админу.
- `bot/handlers/orders.py` — список заказов покупателя с пагинацией.
- `bot/handlers/admin.py` — админ-панель: список заказов на подтверждение,
  управление остатками (категории → подкатегории → товары → задать
  остаток), удаление товара, экспорт отчётов.
- `bot/handlers/posts.py` — синхронизация постов канала в БД, обработка
  комментариев с количеством товара.

## FSM (состояния)

Состояние пользователя хранится как строка в `context.user_data['state']`.
Хендлер, который должен показать следующий шаг, выставляет состояние и
ждёт текстового сообщения. **Все** текстовые сообщения ловит
`message_dispatcher` в `fsm_inputs.py` и раскидывает по состояниям
(`process_*` функции). Если добавляешь новый текстовый шаг — регистрируй
его именно там, а не отдельным `MessageHandler` в другом файле (раньше в
`catalog.py` были функции `search_name_result`/`search_article_result`,
которые никогда не вызывались именно по этой причине — их удалили).

`bot/filters.py` с классом `StateFilter` был **не использован вообще**
(маршрутизация по состоянию всегда делалась вручную в
`message_dispatcher`) — файл удалён.

## Кэш каталога (`bot/cache.py` + `bot/db.py`)

Покупатели листают каталог очень часто, а товары меняются редко — поэтому
читающие запросы каталога кэшируются на 60 секунд (`CATALOG_CACHE_TTL` в
`db.py`):

- `get_active_categories(session)` — список категорий с активными товарами.
- `get_active_products_in_category(session, category)` — активные товары
  категории (используется и для группировки по подкатегориям, и для
  пагинации карточек — листать одну категорию теперь не делает лишних
  запросов в БД).
- `get_all_active_products(session)` — все активные товары (поиск по
  названию).

**Важно:** при любом изменении товаров кэш нужно сбрасывать вызовом
`invalidate_catalog_cache()` (импортируется из `bot.db` или `bot.cache`).
Сейчас это сделано во всех местах, где меняются товары:

- `db.py: upsert_product()` — сброс встроен внутрь функции (синхронизация
  из канала, ручная пересинхронизация).
- `fsm_inputs.py: process_admin_set_stock` — ручное изменение остатка.
- `fsm_inputs.py: process_admin_delete_by_articles` — массовое удаление.
- `fsm_inputs.py: porder_confirm` — списание остатка при заказе через
  комментарий к посту.
- `cart.py: checkout_confirm` — списание остатка при оформлении из корзины.
- `checkout.py: payment_cancel` — возврат остатка при отмене.
- `checkout.py: admin_pay_ok` — повторная актуализация is_active/in_stock.
- `admin.py: delete_confirm` — удаление товара админом.

**Если добавляешь новое место, которое создаёт/меняет/удаляет `Product`
(stock, is_active, in_stock), нужно вызвать `invalidate_catalog_cache()`
после commit**, иначе покупатели до минуты будут видеть устаревшие данные
(например, товар, который уже распродан).

Админские экраны управления остатками (`admin.py`) кэш **не** используют —
они всегда читают БД напрямую, потому что админу нужны все товары
(включая скрытые/неактивные) в реальном времени, а нагрузка там низкая
(один пользователь).

## Что было исправлено в этой сессии (история правок)

Контекст: пользователь просил довести хендлеры до продакшен-уровня и
добавить кэширование каталога, без добавления нового функционала.

**Критические баги:**
- `admin.py`: пагинация категорий/подкатегорий/товаров в управлении
  остатками была сломана — `query.data.split(":")[1]` вместо `[-1]`
  (брался кусок `"catlist"`/`"sublist"`/`"prodpage"` вместо номера
  страницы → `ValueError` при `int()`). Исправлено везде на `[-1]`.
- `main.py`: джоба ежедневных напоминаний об оплате использовала
  `from reminders import send_reminders` (без `bot.`) — модуль с таким
  путём не существует, джоба падала каждый день. Исправлено на
  `from bot.reminders import send_reminders`.
- `catalog.py`: кнопка «↩️ К подкатегориям» (callback
  `catalog:back_to_subs`) генерировалась, но обработчик `back_to_subs`
  никогда не регистрировался в `register()` — кнопка не работала у
  покупателей. Регистрация добавлена.
- Markdown-инъекции: свободный текст (адрес доставки, текст обращения к
  админу, имя пользователя/username с `_`/`*`) подставлялся в сообщения с
  `parse_mode="Markdown"` без экранирования → Telegram отбрасывал
  сообщение целиком (`Can't parse entities`), из-за чего админ мог вообще
  не получить уведомление о заказе/обращении. Экранирование
  (`escape_markdown`) добавлено в: `utils.format_order_for_admin`,
  `fsm_inputs.confirm_data_yes`, `fsm_inputs.process_awaiting_address`,
  `fsm_inputs.process_contact_admin`, `orders.orders_list`,
  `admin.confirm_list`, `posts.handle_comment`.
- `fsm_inputs.process_contact_admin`: при ошибке отправки сообщения админу
  пользователю всё равно показывалось «✅ Сообщение отправлено» — ложный
  успех. Теперь при ошибке показывается «❌ не удалось отправить».
- `fsm_inputs.process_admin_sync`: после синхронизации товара всегда
  показывалось «скрыт, остаток 0» независимо от реального остатка/статуса
  видимости. Теперь текст отражает фактический `is_active`/`stock`.
- `admin.py`: возврат к списку товаров после установки остатка всегда
  возвращал на страницу 0 (`admin_current_page` никогда не сохранялся).
  Теперь `show_stock_products_page` сохраняет текущую страницу.

**Безопасность:**
- Удалён `bot/script.py` — debug-скрипт с **хардкодным боевым токеном
  бота**, тем же самым, что в `.env`, и закоммиченным в git-историю.
  **Нужно обязательно отозвать/перевыпустить токен через @BotFather**, он
  считается скомпрометированным.
- Добавлен `.dockerignore` — раньше `COPY . .` в Dockerfile затягивал в
  образ `.env` (секреты), `.git`, `orders.db`, `.venv`.

**Производительность:**
- Кэширование каталога (см. раздел выше) — устраняет повторные одинаковые
  запросы к БД на каждое нажатие кнопки каталога.
- `catalog.show_products_page` теперь делает один кэшированный запрос
  списка товаров категории + фильтрация/пагинация в Python вместо двух
  SQL-запросов (count + offset/limit) на каждую страницу.
- `fsm_inputs.process_search_name` использует кэшированный список активных
  товаров вместо полного сканирования таблицы на каждый поисковый запрос.

**Чистка мёртвого кода:**
- Удалены `catalog.search_name_result` / `catalog.search_article_result` —
  никогда не были подключены как хендлеры (реальная обработка состояний
  `search_name`/`search_article` — в `fsm_inputs.py`).
- Удалён `admin.stock_catpage` — обработчик регистрировался, но ни одна
  кнопка не генерировала его callback (`admin:stock_catpage:...`).
- Удалён `bot/filters.py` (`StateFilter`) — не использовался нигде.
- Убраны неиспользуемые импорты (`Order`, `OrderItem`,
  `get_or_create_user`, `get_or_create_draft`, `add_item_to_order`,
  `quote`/`unquote` и т.п.) в `catalog.py`, `admin.py`, `cart.py`,

- Добавлен `bot/__init__.py` (раньше пакет работал только за счёт
  implicit namespace packages — это рабочий, но менее надёжный вариант).

## Что осталось как есть (сознательно не трогали)

- **Время напоминаний**: в `main.py` комментарий говорит «09:00 МСК», а
  код использует `time(hour=6, minute=0, tzinfo=Europe/Moscow)` — то есть
  фактически 6:00 МСК. Это явное расхождение комментария и кода, но
  поменять час — бизнес-решение, а не баг; пользователю нужно явно
  спросить/сказать, какое время реально нужно.
- `with_for_update()` в `cart.py` при списании остатков — на SQLite это
  no-op (диалект SQLite не поддерживает `FOR UPDATE`, SQLAlchemy просто
  игнорирует). Под нагрузкой в проде на SQLite возможны редкие гонки при
  одновременном заказе последней единицы товара. При переезде на Postgres
  (`DATABASE_URL` уже поддерживает `asyncpg`) это начнёт реально работать
  как блокировка строки.
- `requirements.txt` без фиксированных версий (`>=20.0` и без версий
  вообще для остальных) — для продакшена стоит закрепить версии
  (`pip freeze` после тестового прогона), чтобы будущий `pip install`
  не подтянул breaking-changes из новой мажорной версии
  python-telegram-bot.
- `fsm_inputs.py` импортирует `func` из sqlalchemy, но не использует —
  безобидный мёртвый импорт, был в коде до этой сессии, не тронут, чтобы
  не плодить лишние правки.

## Переменные окружения (.env)

```
BOT_TOKEN=            # токен от @BotFather — СЕЙЧАС СКОМПРОМЕТИРОВАН, ПЕРЕВЫПУСТИТЬ
ADMIN_USER_ID=        # Telegram ID администратора
ADMIN_CHAT_ID=        # куда слать уведомления (обычно = ADMIN_USER_ID)
CHANNEL_ID=           # ID канала с товарами (начинается с -100)
DISCUSSION_GROUP_ID=  # ID группы обсуждения канала (для комментариев)
DATABASE_URL=         # sqlite+aiosqlite:///orders.db или postgres+asyncpg://...
```

## Деплой

`Dockerfile` собирает образ на `python:3.12-slim`, ставит зависимости,
копирует код, запускает `python -m bot.main`. `.dockerignore` теперь
исключает секреты/БД/git/venv из образа — не убирай его.
